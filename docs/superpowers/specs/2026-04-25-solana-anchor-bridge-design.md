## 1. Context

This is the Solana side of a four-person student bridge project that moves a single ERC-20 from Ethereum Sepolia to Solana Devnet via a single trusted relayer.

Team split:
- **Aaron** — Solana/Rust Anchor program (this spec)
- **Yang** — Ethereum/Solidity contracts (`TestToken.sol`, `EthBridge.sol`)
- **Leah & Harry** — TypeScript relayer + system architecture (already implemented in `src/`)

The system is intentionally a **trusted, validator-style bridge**: Solana does not independently verify Ethereum state. It only verifies that an authorized relayer submitted a syntactically valid message bound to the configured source domain.

The team's joint report (`Blockchain 4.8.pdf`) describes the full system in detail. This spec defines what the Solana side actually implements and how it diverges from / pins down the report's higher-level description.

## 2. Scope

**In scope:**
- Anchor program (`programs/bridge`) implementing `initialize`, `mint_wrapped`, `set_paused`, `set_relayer`.
- `BridgeConfig` singleton PDA and `ProcessedMessage` per-message PDA.
- Wrapped SPL mint with PDA mint authority.
- 147-byte fixed-width binary `BridgePayload` (matches the report).
- `anchor test` suite with at least 10 named tests covering happy path, auth, replay, pause, and source-binding.
- Devnet deployment: `anchor deploy`, create wrapped mint, transfer mint authority to PDA, initialize `BridgeConfig`.
- Coordination handoff: regenerated IDL + a one-function change for the relayer's `BridgePayload` builder.
- Writeup: a standalone "Solana Aaron" section (~4 pages) for the joint PDF.

**Out of scope:**
- Reverse path (SOL → ETH burn/unlock).
- k-of-n multi-relayer attestation (analyzed in the report's extension section, not implemented).
- Wormhole VAA verification.
- Token-2022, multiple wrapped assets, mainnet deployment.
- Full E2E demo through Yang's live Sepolia contract (the spec leaves room for it but does not require it).

## 4. Account model

Two account types, mirroring Solana's stateless-program convention.

### 4.1 `BridgeConfig` (singleton)

PDA seeds: `["config"]`. Initialized once by `initialize`, then mutated only by `set_paused` / `set_relayer`.

```rust
#[account]
pub struct BridgeConfig {
    pub admin: Pubkey,                  // 32 — controls pause + relayer rotation
    pub authorized_relayer: Pubkey,     // 32 — only signer accepted by mint_wrapped
    pub source_chain_id: u16,           //  2 — Sepolia: 11155111 truncated? See §4.4
    pub source_bridge: [u8; 20],        // 20 — EthBridge.sol contract address
    pub source_token: [u8; 20],         // 20 — bound source ERC-20 address
    pub wrapped_mint: Pubkey,           // 32 — destination SPL mint
    pub mint_authority_bump: u8,        //  1 — cached bump for ["mint_authority"]
    pub config_bump: u8,                //  1 — cached bump for ["config"]
    pub paused: bool,                   //  1
    // total payload: 141 bytes; account size = 8 (discriminator) + 141 = 149
}
```

### 4.2 `ProcessedMessage` (per-message replay key)

PDA seeds:
```
["processed",
 source_chain_id.to_le_bytes(),    // 2 bytes
 source_bridge,                    // 20 bytes
 nonce.to_le_bytes()]              // 8 bytes
```

The seeds are **domain-separated** rather than just `["processed", nonce]`. Including the source chain and bridge address means future expansion (multiple source chains or upgraded source contracts) does not require migrating existing replay state — message identity is `(source_chain, source_bridge, nonce)` triplet, not just `nonce`.

```rust
#[account]
pub struct ProcessedMessage {
    pub nonce: u64,                     //  8
    pub source_tx_hash: [u8; 32],       // 32
    pub source_log_index: u32,          //  4
    pub amount: u64,                    //  8
    pub recipient: Pubkey,              // 32
    pub processed_slot: u64,            //  8
    // total payload: 92 bytes; account size = 8 + 92 = 100
}
```

This account exists purely so its PDA can't be re-derived. The data fields are an audit trail for off-chain inspection; they are never read by the program after creation.

### 4.3 Mint authority PDA

PDA seeds: `["mint_authority"]`. Has no account data — it is only used as the SPL `Mint`'s `mint_authority` field and as a CPI signer.

After deploying the program and creating the wrapped SPL mint, a one-time CLI step calls SPL `set_authority(MintTokens, <pda>)` to transfer mint authority from the human deployer to this PDA. From that moment, only this program can mint wrapped tokens.

### 4.4 `source_chain_id` choice

Sepolia's EVM chain ID is `11155111`, which does not fit in `u16`. The report uses a custom 2-byte "domain" identifier (Wormhole-style), not the EVM chain ID. We adopt the same convention:

| Domain | Value |
|---|---|
| Sepolia (this prototype) | `1` |
| (reserved for future Ethereum-mainnet) | `2` |

The relayer must encode the same value in `payload.source_chain_id`, and `BridgeConfig.source_chain_id` is initialized to `1`. A mismatch becomes an `InvalidSource` error.

## 5. Bridge payload (147 bytes, fixed-width binary)

This replaces the current string-based `BridgePayload` in `idl/bridge.json`.

```rust
#[derive(AnchorSerialize, AnchorDeserialize, Clone, Debug)]
pub struct BridgePayload {
    pub version: u8,                    //  1 — must equal 1
    pub source_chain_id: u16,           //  2
    pub source_bridge: [u8; 20],        // 20
    pub source_token: [u8; 20],         // 20
    pub nonce: u64,                     //  8
    pub amount: u64,                    //  8 — relayer rejects ETH amounts > u64::MAX
    pub sender_eth: [u8; 20],           // 20
    pub recipient_sol: [u8; 32],        // 32
    pub source_tx_hash: [u8; 32],       // 32
    pub source_log_index: u32,          //  4
    // total: 147 bytes
}
```

All multi-byte integers are little-endian (Borsh default).

### 5.1 Why 147-byte binary instead of strings

- Smaller compute and rent footprint than string parsing.
- 32-byte Solana pubkeys and 20-byte EVM addresses are canonically binary; encoding them as strings introduces ambiguity (base58 vs. hex vs. checksummed).
- Matches what real Wormhole VAAs look like, which is the comparison the report's evaluation rests on.
- Forces the `amount` overflow check at the relayer boundary (ETH `uint256` does not fit in `u64`), making the type boundary explicit instead of silently truncating.

### 5.2 Coordination cost (relayer)

Leah & Harry's `src/submitter.ts` currently builds:

```ts
const bridgePayload = {
  nonce: new BN(msg.nonce),
  sender: msg.sender,                // string
  recipient: msg.recipient,          // string
  ...
};
```

This needs to become:

```ts
const bridgePayload = {
  version: 1,
  sourceChainId: 1,
  sourceBridge: hexToBytes(config.ETH_BRIDGE_ADDRESS),    // [u8; 20]
  sourceToken: hexToBytes(config.ETH_TOKEN_ADDRESS),      // [u8; 20]
  nonce: new BN(msg.nonce),
  amount: new BN(msg.amount),                              // throws if > u64::MAX
  senderEth: hexToBytes(msg.sender),                      // [u8; 20]
  recipientSol: bs58.decode(msg.recipient),               // [u8; 32]
  sourceTxHash: hexToBytes(msg.sourceTxHash),             // [u8; 32]
  sourceLogIndex: msg.logIndex,
};
```

It is a one-function change, plus adding `ETH_TOKEN_ADDRESS` to the relayer's `.env`. The IDL regenerated by `anchor build` will give them autocomplete in TypeScript.

## 6. Instructions

| Instruction | Signer | Args | Mutates |
|---|---|---|---|
| `initialize` | admin (one-time) | `InitializeArgs` | creates `BridgeConfig` |
| `mint_wrapped` | authorized_relayer | `BridgePayload` | creates `ProcessedMessage`, mints SPL tokens |
| `set_paused` | admin | `bool` | `BridgeConfig.paused` |
| `set_relayer` | admin | `Pubkey` | `BridgeConfig.authorized_relayer` |

### 6.1 `initialize`

```rust
pub struct InitializeArgs {
    pub authorized_relayer: Pubkey,
    pub source_chain_id: u16,
    pub source_bridge: [u8; 20],
    pub source_token: [u8; 20],
    // wrapped_mint and mint_authority bump are derived from accounts
}
```

Accounts:
- `admin` (signer, payer)
- `config` (PDA `["config"]`, init, payer = admin)
- `wrapped_mint` (the mint whose authority is the program PDA; not mutated here)
- `mint_authority` (PDA `["mint_authority"]`; only used to compute and store its bump)
- `system_program`

Body:
1. Verify `wrapped_mint.mint_authority == Some(mint_authority_pda)`. If not, fail with `MintAuthorityNotTransferred`. (This means the operator forgot the `set_authority` step.)
2. Populate `BridgeConfig` fields, store both bumps, set `paused = false`.

### 6.2 `mint_wrapped`

This is the only complex instruction. Accounts:

```
relayer              signer, mut (pays rent for ProcessedMessage)
config               PDA ["config"], readonly
processed_message    PDA ["processed", source_chain_id_le, source_bridge, nonce_le], init
wrapped_mint         mut (supply increases)
mint_authority       PDA ["mint_authority"], readonly (CPI signer)
recipient            readonly (the Solana wallet receiving wrapped tokens)
recipient_token_account   mut (must equal ATA(recipient, wrapped_mint))
token_program        SPL Token
system_program
```

Body (in order — order matters for clean errors):

```rust
require!(!config.paused, BridgeError::Paused);
require!(relayer.key() == config.authorized_relayer, BridgeError::UnauthorizedRelayer);
require!(payload.version == 1, BridgeError::UnsupportedVersion);
require!(payload.source_chain_id == config.source_chain_id, BridgeError::InvalidSource);
require!(payload.source_bridge == config.source_bridge, BridgeError::InvalidSource);
require!(payload.source_token == config.source_token, BridgeError::InvalidSource);
require!(payload.amount > 0, BridgeError::ZeroAmount);
require!(wrapped_mint.key() == config.wrapped_mint, BridgeError::WrongMint);

// Recipient ATA check (Anchor account constraint will also enforce this,
// but we re-check explicitly so the error is unambiguous)
let expected_ata = get_associated_token_address(&payload.recipient_sol.into(), &wrapped_mint.key());
require!(recipient_token_account.key() == expected_ata, BridgeError::InvalidRecipientAta);
require!(recipient.key() == Pubkey::from(payload.recipient_sol), BridgeError::RecipientMismatch);

// Replay protection: ProcessedMessage uses `init` (not init_if_needed).
// A second submission for the same (source_chain_id, source_bridge, nonce) fails here
// at account creation. We do not need an explicit MessageAlreadyProcessed check.

// CPI mint
let mint_authority_seeds: &[&[u8]] = &[b"mint_authority", &[config.mint_authority_bump]];
token::mint_to(
    CpiContext::new_with_signer(
        token_program.to_account_info(),
        token::MintTo {
            mint: wrapped_mint.to_account_info(),
            to: recipient_token_account.to_account_info(),
            authority: mint_authority.to_account_info(),
        },
        &[mint_authority_seeds],
    ),
    payload.amount,
)?;

// Populate ProcessedMessage as audit record
processed_message.set_inner(ProcessedMessage {
    nonce: payload.nonce,
    source_tx_hash: payload.source_tx_hash,
    source_log_index: payload.source_log_index,
    amount: payload.amount,
    recipient: recipient.key(),
    processed_slot: Clock::get()?.slot,
});
```

**ATA creation strategy:** the relayer creates the recipient ATA in a prior instruction in the same Solana transaction (using `createAssociatedTokenAccountIdempotentInstruction`). The bridge program does not create ATAs itself — this keeps the bridge instruction smaller and avoids `init_if_needed` (which carries a [known footgun](https://github.com/coral-xyz/anchor/issues) around re-initialization attacks if not guarded carefully). Acceptable because both instructions share the same atomic transaction.

### 6.3 `set_paused`

```rust
require!(admin.key() == config.admin, BridgeError::UnauthorizedAdmin);
config.paused = new_state;
```

### 6.4 `set_relayer`

```rust
require!(admin.key() == config.admin, BridgeError::UnauthorizedAdmin);
config.authorized_relayer = new_relayer;
```

No "old relayer must not have pending messages" check — pending messages were minted by signature anyway, and rotation is about future submissions.

## 7. Errors

```rust
#[error_code]
pub enum BridgeError {
    #[msg("Bridge is paused")]                          Paused,
    #[msg("Signer is not the authorized relayer")]      UnauthorizedRelayer,
    #[msg("Signer is not the admin")]                   UnauthorizedAdmin,
    #[msg("Unsupported payload version")]               UnsupportedVersion,
    #[msg("Source domain mismatch")]                    InvalidSource,
    #[msg("Amount must be non-zero")]                   ZeroAmount,
    #[msg("Wrapped mint does not match configured mint")] WrongMint,
    #[msg("Recipient ATA does not match")]              InvalidRecipientAta,
    #[msg("Recipient pubkey does not match payload")]   RecipientMismatch,
    #[msg("Mint authority not transferred to bridge PDA")] MintAuthorityNotTransferred,
}
```

Note there is no explicit `MessageAlreadyProcessed`. Anchor's account-init failure on the `ProcessedMessage` PDA serves as the replay error, which appears in client logs as "account already in use" — clear enough for an audit trail and one fewer code path to maintain. The writeup documents this choice.

| 4 | `mint_wrapped_unauthorized_relayer_rejected` | Different signer fails with `UnauthorizedRelayer` |
| 5 | `mint_wrapped_paused_rejected` | After `set_paused(true)`, mint fails with `Paused` |
| 6 | `mint_wrapped_wrong_source_bridge_rejected` | `InvalidSource` |
| 7 | `mint_wrapped_wrong_source_token_rejected` | `InvalidSource` |
| 8 | `mint_wrapped_wrong_chain_id_rejected` | `InvalidSource` |
| 9 | `mint_wrapped_zero_amount_rejected` | `ZeroAmount` |
| 10 | `mint_wrapped_unsupported_version_rejected` | `UnsupportedVersion` |
| 11 | `mint_wrapped_duplicate_nonce_rejected` | Second submit fails at PDA init (replay) |
| 12 | `mint_wrapped_wrong_recipient_ata_rejected` | `InvalidRecipientAta` |

## 9. Devnet deployment

`migrations/deploy.ts` orchestrates the post-`anchor deploy` setup. Manual order:

1. `solana config set --url devnet`
2. `anchor build` → captures the program ID.
3. `anchor deploy` → uploads BPF binary; this may need to run twice if the deployer account needs more SOL.
4. `npx ts-node migrations/deploy.ts` does:
   - Create new SPL mint with decimals = 6 (matches a typical USDC-style demo token).
   - Compute `mint_authority` PDA from the deployed program ID.
   - `setAuthority(mint, MintTokens, mintAuthorityPda)`.
   - Call `program.methods.initialize(...)` with admin, relayer pubkey, source_chain_id = 1, source_bridge bytes, source_token bytes.
   - Print: program ID, config PDA, wrapped mint, mint authority PDA, relayer pubkey.

Those printed values are what Leah & Harry put in their `.env`:

```
SOL_PROGRAM_ID=<from step 4>
SOL_WRAPPED_MINT=<from step 4>
SOL_RELAYER_KEYPAIR=<existing>
ETH_TOKEN_ADDRESS=<NEW — from Yang's deployment>
```

Funding: relayer needs ~0.5 SOL Devnet to comfortably cover ATA-creation rent (~0.002 SOL per recipient) and tx fees over a demo session.

## 10. Writeup ("Solana Aaron" section, ~4 pages, ~2000 words)

Lives in the team's joint PDF where the empty "Solana Aaron" page currently sits.

Section outline:

1. **What I built** (~200 words). Scope statement: the Anchor program, `BridgeConfig`, `ProcessedMessage`, mint authority PDA, the `mint_wrapped` instruction, deployment scripts, tests. What I did *not* build (relayer, ETH side, k-of-n).

2. **Account model** (~400 words). Why two PDAs and not three or one. Why `["config"]` for the singleton. Why domain-separated seeds for `ProcessedMessage` (future-proofing message identity). Why a separate mint authority PDA instead of using `BridgeConfig` itself as the authority (separation of concerns: config is mutable, mint authority must never be).

3. **The mint_wrapped instruction** (~500 words). Walk through validation order. Explain why each check exists and what attack each prevents. The choice to enforce the recipient ATA at the program level rather than trusting the relayer.

4. **Replay protection** (~300 words). Why `init` (not `init_if_needed`) on `ProcessedMessage` is the actual replay defense, leveraging Solana account-creation atomicity. Why I omitted an explicit `MessageAlreadyProcessed` error code. The argument that this works *because* Solana transactions are atomic — a failed mint cannot leave a "processed" record without tokens.

5. **Token integration** (~250 words). Token Program vs. Token-2022 — chose legacy for Anchor TS client compatibility. ATA handling on the relayer side, with one paragraph on the `init_if_needed` decision. CPI signer seeds.

6. **Tests** (~200 words). The 14 tests, what each catches. The duplicate-nonce test as the load-bearing one.

7. **Devnet deployment notes** (~150 words). The mint-authority transfer step as the deployment gotcha. Values handed to the relayer team.

8. **What I'd change with more time** (~100 words). On-chain ATA init via `init_if_needed` with proper guards. Pre-flight balance check. Migration to k-of-n threshold (referencing the report's extension analysis).

## 12. Open risks

| Risk | Mitigation |
|---|---|
| Anchor 0.30 IDL format breaks the relayer's import of `idl/bridge.json` | Test with the relayer team after first `anchor build` — fix once. |
| Devnet rent for `ProcessedMessage` accounts piles up over many test runs | These accounts are 100 bytes; trivial. Acceptable for a student project. |
| Relayer key compromise → malicious mint | This is the trusted-bridge limitation the report acknowledges. `set_paused` is the only mitigation. |
| `source_chain_id` discriminator collision with future Wormhole-style integrations | We use `1` for Sepolia, deliberately not the EVM chain ID. Documented above. |
